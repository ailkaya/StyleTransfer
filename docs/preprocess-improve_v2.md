下面是一份可以直接交付给工程/数据团队执行的**风格可控微调数据构建规范（鲁迅风示例）**。目标是：用约 500 条原始素材，构造一个**可泛化到任意任务、且风格可控**的训练集，用于 QLoRA/LoRA 微调。

---

# 风格可控微调数据构建规范（v1.0）

## 0. 目标与范围

**目标**

* 支持：写作 / 解释 / 总结 / 改写 / 续写 五类任务
* 支持：显式风格控制（如“用鲁迅风格…”）
* 支持：风格强度调节（强 / 弱 / 普通）
* 支持：不同输入形态（短指令 / 长上下文 / 多轮对话）
* 输出：统一为 chat 格式（messages）

**非目标**

* 不追求复现具体作品
* 不做事实知识注入（非重点）

---

## 1. 数据总体结构

统一格式：

```json
{
  "messages": [
    {"role": "system", "content": "<风格控制或空>"},
    {"role": "user", "content": "<任务描述 + 输入>"},
    {"role": "assistant", "content": "<目标输出>"}
  ]
}
```

---

## 2. 任务类型设计（至少5类）

### 2.1 写作（Generation）

**模板**

```json
{
  "user": "用鲁迅风格写一段关于{topic}",
  "assistant": "..."
}
```

**变体**

* 自由写作
* 议论 / 描写 / 叙事
* 指定情绪（讽刺、冷峻等）

---

### 2.2 解释（Explanation）

```json
{
  "user": "用鲁迅风格解释什么是{concept}",
  "assistant": "..."
}
```

要求：

* 保留解释逻辑
* 风格不影响信息正确性

---

### 2.3 总结（Summarization）

```json
{
  "user": "用鲁迅风格总结以下内容：\n{long_text}",
  "assistant": "..."
}
```

要求：

* 保留关键信息
* 控制压缩率（30%~60%）

---

### 2.4 改写（Rewrite）

```json
{
  "user": "把下面这段话改写成鲁迅风格：\n{text}",
  "assistant": "..."
}
```

关键：

* 语义保持
* 表达变化

---

### 2.5 续写（Continuation）

```json
{
  "user": "用鲁迅风格续写这段话：\n{prefix}",
  "assistant": "..."
}
```

要求：

* 风格连续
* 语气一致
* 不重复前文

---

## 3. 风格显式化（必须）

### 3.1 推荐方式（优先）

在 user 中显式指定：

```text
用鲁迅风格...
以鲁迅的笔调...
模仿鲁迅语气...
```

---

### 3.2 可选增强（system）

```json
{
  "role": "system",
  "content": "你是一位擅长模仿鲁迅文风的写作助手..."
}
```

建议：

* 不要 100% 使用 system（避免过拟合）
* 占比 ≤ 50%

---

## 4. 风格强弱控制（关键）

### 4.1 定义

| 类型  | 特征                    |
| --- | --------------------- |
| 强鲁迅 | 明显句式（“我大抵…”、“却…”）、强节奏 |
| 弱鲁迅 | 保留语气，但减少标志性句式         |
| 普通  | 无明显风格                 |

---

### 4.2 数据配比（建议）

```text
强鲁迅：50%
弱鲁迅：30%
普通：20%
```

---

### 4.3 示例

**强鲁迅**

```text
我大抵是见惯了这些事的……
```

**弱鲁迅**

```text
这类事情，大约也并不稀奇……
```

**普通**

```text
这种情况其实很常见。
```

---

## 5. 输入多样性设计

### 5.1 短指令（Short Instruction）

```text
写一段关于孤独
解释HTTP
总结这段话
```

---

### 5.2 长上下文（Long Context）

```text
请总结以下内容：
（300~800字文本）
```

---

### 5.3 多轮对话（Multi-turn）

```json
{
  "messages": [
    {"role": "user", "content": "用鲁迅风格写一段关于城市"},
    {"role": "assistant", "content": "..."},
    {"role": "user", "content": "再写得更讽刺一点"},
    {"role": "assistant", "content": "..."}
  ]
}
```

要求：

* 保持上下文一致
* 风格递进

---

## 6. 数据构造流程（Pipeline）

### Step 1：原始语料切分

* 来源：鲁迅小说
* 粒度：句 / 短段（20~100字）

---

### Step 2：语义抽象

为每段生成：

* 描述性 prompt（“表达XX情绪”）
* 或任务化 prompt（解释/总结等）

---

### Step 3：多样化生成

每条原始数据扩展为：

| 类型   | 数量  |
| ---- | --- |
| 强风格  | 1   |
| 弱风格  | 1   |
| 普通   | 1   |
| 任务变体 | 1~2 |

👉 每条原始 → 3~5 条训练数据

---

### Step 4：任务映射

同一语料生成多任务版本：

* 写作
* 改写
* 续写
* 总结

---

### Step 5：混入非鲁迅数据（必须）

占比：20%

来源：

* 普通文本
* 其他风格

---

## 7. 数据规模建议

如果原始 500 条：

```text
500 × 4 = 2000（扩展）
+ 500（非鲁迅）
= 2500 条训练数据
```

---

## 8. 质量控制（必须执行）

### 8.1 自动规则

* 去重（文本相似度 > 0.9 删除）
* 长度过滤（10~512 token）
* 非法字符过滤

---

### 8.2 人工抽检（至少5%）

检查：

* 风格是否一致
* 语义是否正确
* 是否模板化

---

## 9. 常见错误（必须避免）

### ❌ 错误1：全部强风格

→ 导致风格塌缩

---

### ❌ 错误2：只有写作任务

→ 无法泛化到解释/总结

---

### ❌ 错误3：一对一映射

→ 模型记忆而非学习

---

### ❌ 错误4：无续写数据

→ 无法做 continuation

---

## 10. 最小可用版本（MVP）

如果资源有限，只做：

1. 加“用鲁迅风格”前缀
2. 加入 30% 非写作任务
3. 每条数据做 2 个改写版本
4. 增加 20% 普通风格

👉 即可显著提升泛化能力

---

# 总结（给工程团队的核心原则）

> **风格必须是“条件”，不是“默认”。**
> **数据必须覆盖“任务 × 风格 × 表达”的组合空间。**
